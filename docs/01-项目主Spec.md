# 项目主 Spec（个人博客 / 个人站点）

本文档定义本仓库的**产品目标、范围边界、信息架构与变更规则**，属于 **第二层（外部 Spec）**：只保留高层约束，**代码级细节放在第一层（源码内联）**，见下文「文档三层与归档」。

技术索引与 Skill 入口见 [02-全局Reference规范.md](./02-全局Reference规范.md)。

## 文档三层与归档

### 第一层：源码内联 Spec（与代码强绑定）

- **载体**：公共 API / 组件 Props 的 TSDoc、**业务常量与配置**旁的注释、`nuxt.config` 等关键配置处的简短说明。
- **原则**：能写进代码边上的不写进外部长文；**改行为时同步改注释**，避免脱节。
- **示例路径**：`app/constants/site.ts`、`app/constants/favoriteSkills.ts`、各页面/组件内对非显然逻辑的注释。

### 第二层：外部 Spec（本项目 + 模块，高层约束）

- **载体**：[01-项目主Spec.md](./01-项目主Spec.md)（本文）、[02-全局Reference规范.md](./02-全局Reference规范.md)、[03-模块拆解规划.md](./03-模块拆解规划.md)、[`docs/modules/*.md`](./modules/README.md)。
- **原则**：**业务边界、信息架构、交互与验收口径**；不抄实现、不贴大段源码。
- **`project-dev-conventions` Skill**：供人/Agent 快速对齐习惯；与第一层真实配置冲突时，**以代码与第一层为准**，并回写 Skill 或本文。

### 第三层：历史归档 Spec（只读快照）

- **载体**：[./archive/](archive/README.md) 下按版本/日期区分子目录。
- **原则**：里程碑封版时**复制**当时相关文档进归档；**不修改**已有归档内容，后续只新增新版本目录。

### 何时必须更新文档（简表）

| 变更类型 | 第一层（源码） | 第二层（docs） | 第三层（归档） |
|----------|----------------|----------------|----------------|
| 改路由 / 导航 / 站点级边界 | `site.ts`、`nav` 相关注释 | 更新 **01、03** | 仅封版时新建快照 |
| 某模块行为或验收变化 | 相关组件/常量注释 | 更新 **`docs/modules/...`** 或 **03** | 封版时快照 |
| 技术栈 / ssr 策略锁定变化 | `nuxt.config` 注释或 package 说明 | 更新 **01**、必要时 **02** | 封版时快照 |
| 纯文案 / i18n | `messages` 键旁注释（若需） | 一般不改 01；模块验收若涉及则改模块 Spec | 否 |
| 里程碑发布 | 可选在常量或 README 注版本 | 确认 01～03 与代码一致 | **新建** `archive/...` 快照 |

## 1. 项目定位与核心目标

- **定位**：面向访客的个人站点，以**自我介绍**为主，并预留**项目展示**、**博客**、**收藏（Skills 等）**等栏目；风格简洁、可读、易维护。
- **核心目标**
  - 清晰传达作者身份与简介（首页）。
  - 导航与信息结构一目了然（顶栏 + 固定路由）。
  - 支持 **中文 / 英文** 切换（与现有 i18n 实现一致）。
- **非目标（当前阶段）**
  - 不做独立 CMS、评论系统、用户账号体系。
  - 不做「设计系统级」的 Figma 全量 Token 固化（见 Reference 中设计 Token 占位说明）。
  - 不引入第二套 CSS 框架或与主 Spec 冲突的技术栈（技术锁定见第 4 节）。

## 2. 业务范围与功能边界

### 2.1 应支持

- 首页：站点名、头像、简介文案、关于段落。
- 栏目页（可为占位，逐步充实）：`/project`、`/blogs`、`/favorites`。
- 全局：顶栏导航、外链 GitHub、语言切换与辅助浮层（以现有实现为准）。
- 收藏页：按模块展示结构化列表（当前为 **Skills** 模块，数据可维护在 `app/constants/favoriteSkills.ts`）。

### 2.2 明确不做（除非先更新本 Spec）

- 服务端渲染形态变更：当前 **`nuxt.config` 中 `ssr: false`**，若改为 SSR/混合渲染须先更新本 Spec 并评估部署方式。
- 为极端设备、罕见浏览器单独分支，除非产品明确要求。
- 在实现阶段将「全量 lint / 生产构建」作为 AI 编码会话的常规步骤（人类日常与提交约定见 [02-全局Reference规范.md](./02-全局Reference规范.md#3-编码与-agent-协作)；Agent 行为见 `dev-no-lint-during-coding` Skill）。

## 3. 信息架构与路由

| 路径 | 说明 |
|------|------|
| `/` | 首页（简介 + 关于） |
| `/project` | 项目（当前可为占位） |
| `/blogs` | 博客（当前可为占位） |
| `/favorites` | 收藏（Skills 等模块） |

顶栏导航顺序与路由常量保持一致：`app/constants/site.ts` 中 `navRoutes`。

## 4. 技术栈锁定（摘要）

以下仅为**决策级摘要**，细则见 `.cursor/skills/project-dev-conventions/SKILL.md`。

- **框架**：Nuxt **4.x**（以 [package.json](../package.json) 为准）。
- **UI / 样式**：**仅 Tailwind CSS**（utility / 必要时 `@apply`），全局入口 `app/assets/css/main.css`。
- **代码质量**：ESLint 使用 `@antfu/eslint-config` 与仓库内 Nuxt Flat 配置；**人类提交**时若有 Git 钩子，以 [package.json](../package.json) 为准（摘要见 [02](./02-全局Reference规范.md#3-编码与-agent-协作)）。
- **运行模式**：当前为 **纯客户端渲染（`ssr: false`）**，部署与 SEO 策略需与此一致或经 Spec 变更后同步。
- **字体**：Google Fonts 引入 Borel、DM Mono，Tailwind 扩展 `font-borel`、`font-dm-mono`（见 `nuxt.config.ts`）。

## 5. 非功能约束（轻量）

- **可访问性**：关键交互具备可辨识焦点与语义结构；外链使用 `rel="noopener noreferrer"` 等安全属性（与现有组件一致）。
- **国际化**：`html lang` 随当前语言在布局层设置（见 `app/layouts/default.vue`）。
- **性能**：避免无必要的重依赖与过大首屏资源；具体指标不在此空喊，随上线环境再补验收项。

## 6. 变更与同步规则（SDD 对齐）

1. **无 Spec 不开发**：新功能、路由级改动、技术栈变更须有本 Spec 或 [子模块 Spec](./modules/README.md) 依据。
2. **需求变更先改文档**：先更新**第二层**（本 Spec、02、03 或 `docs/modules/`），再改设计与代码；**第一层**注释与常量随代码提交一并更新。
3. **与代码一致性**：路由、导航、`ssr`、i18n 键等以仓库实现为准；若实现与本文冲突，**有意变更**时须回写本文与相关模块 Spec。
4. **禁止**：在第二层文档中堆叠可维护的代码片段替代第一层；必要时用**文件路径 + 一句话**指向源码。

## 7. 待决问题（随迭代填写）

- 博客与项目栏目的最终内容形态（Markdown 文件、远程数据源、静态列表等）。
- 生产部署域名与是否需要为 SEO 调整 `ssr` 策略。
